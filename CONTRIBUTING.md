# Contributing Guidelines

Thank you for taking the time to contribute! \
To keep the project organized and easy to maintain, please follow this guidelines.

---

## Issue Reporting

When creating an issue, please use the following title format:

```
[TYPE] Short, clear description
```

### Allowed Types

* `[BUG]` – Something is broken or not working as expected
* `[FEAT]` – A new feature or functionality
* `[ENH]` – Improvement to an existing feature
* `[DOC]` – Documentation updates or fixes
* `[REFACTOR]` – Code changes without modified behavior
* `[TEST]`  – Adding or updating tests
* `[CHORE]` – Maintenance tasks (e.g., dependencies, tooling, documentation)

### Examples

* `[BUG] Scene enters an infinite loop`
* `[FEAT] Add inventory system`
* `[ENH] Improve database query performance`
* `[DOC] Update setup instructions`

### Guidelines

* Keep titles concise and descriptive
* Avoid vague titles like “Fix issue” or “Something broken”
* Provide detailed descriptions in the issue body (steps to reproduce, expected behavior, etc.)

---

## Pull Requests

### Target Branch

All pull requests **must be opened against the `devel` branch**.

* Do **not** open pull requests directly to `main`
* The `main` branch is reserved for stable releases only

### General Rules

* Ensure your branch is up to date with `devel` before opening a PR
* Link related issues when applicable (e.g., `Closes #123`)
* Keep pull requests focused and small when possible
* Add or update tests if your changes affect functionality

---

## Code Style & Quality

* Follow the existing code style and conventions
* Write clear, readable, and maintainable code
* Add comments where necessary (but prefer self-explanatory code)

---

Thanks again for contributing!
